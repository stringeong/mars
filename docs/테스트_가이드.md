# M.A.R.S 테스트 가이드

서버(FastAPI)와 워커(Python)의 pytest 스위트가 무엇을 검증하는지 정리한 문서.
총 **132개** (서버 92, 워커 40). 모든 테스트는 외부 의존성 없이 돈다 —
DB는 인메모리 SQLite, Ollama는 mock, 파일시스템은 pytest `tmp_path`를 쓴다.

## 실행 방법

```bash
# 서버
cd server
pip install -r requirements-dev.txt
pytest                       # 전체
pytest tests/test_dag.py     # 파일 하나
pytest -k "diamond" -v       # 이름으로 골라서

# 워커
cd worker
pip install -r requirements-dev.txt
pytest
```

CI(GitHub Actions, `.github/workflows/ci.yml`)는 매 PR과 main 푸시마다
서버 pytest / 워커 pytest / 웹 `tsc` 타입체크+vite 빌드 3개 잡을 실행한다.

## 서버 테스트 상세

### test_dag.py — DAG 검증 (요구사항 2.5)

`validate_graph`가 "순환 구조나 실행 불가능한 구성은 검증 단계에서 차단한다"는
계약을 지키는지 확인한다. 생성기(LLM 출력 검증)와 서비스 저장·실행 시작이
모두 이 함수에 의존하므로 가장 먼저 지켜야 할 방어선이다.

- **차단해야 하는 것**: 빈 그래프, 중복 노드 id, 존재하지 않는 노드를 가리키는
  간선, 자기 자신으로의 간선, 2노드 순환, 긴 순환, 분리된 컴포넌트 중 하나에만
  있는 순환.
- **허용해야 하는 것**: 노드 1개짜리 그래프, 분리된 컴포넌트, 중복 간선(무시).
- **위상 정렬**: 반환된 순서에서 모든 간선의 source가 target보다 앞에 오는지.
- `parents_of`(부모 목록, 중복 간선 dedupe)와 `terminal_nodes`(나가는 간선이
  없는 노드 = 최종 결과 수집 대상) 계산.

### test_orchestrator.py — 실행 생명주기

시스템에서 가장 복잡한 상태 전이 로직. `blocked → ready → running → done|failed`
전이와 실행 단위 상태를 다이아몬드 그래프(a→b, a→c, b→d, c→d)까지 포함해 검증한다.

- **작업 생성**: 선행이 없는 노드만 `ready`, 나머지는 `blocked`. 노드의
  role_prompt/model/allowed_folders가 TaskRecord로 복사되는지.
- **기기 온라인 판정**: 하트비트 없음 → 오프라인, 최근 → 온라인,
  `HEARTBEAT_TIMEOUT_SEC` 초과 → 오프라인.
- **끊긴 기기 작업 회수 (UC-204 e203)**: 하트비트가 끊긴 기기의 `running` 작업이
  `ready`로 돌아가 다른 기기가 가져가는지. 온라인 기기의 작업은 건드리지 않는지.
- **작업 선점**: 두 기기가 순차로 폴링해도 같은 작업을 받지 않는지(원자적
  UPDATE), ready가 없으면 `None`, **다른 사용자의 작업은 절대 할당되지 않는지**.
- **입력 컨텍스트 조립**: `[사용자 실행 요청]` + 부모 출력들이 부모 순서대로
  이어지는지, 출력이 빈 부모는 건너뛰는지.
- **완료 전파**: 부모가 둘인 노드는 **둘 다** done이 될 때까지 blocked 유지.
- **결과 통합 (UC-205)**: 전부 done이면 말단 노드 출력만 `## 에이전트명` 형식으로
  `---` 구분자와 함께 합쳐지고 실행이 `completed`가 되는지.
- **실패 전파 (UC-204 e202)**: 한 작업이 실패하면 실행이 `failed`가 되고
  blocked/ready 작업이 "선행 작업 실패로 취소됨"으로 정리되는지.
  ※ 현재 동작 고정: 이미 `running`인 형제는 취소되지 않는다 — 동작을 바꾸면
  `test_failure_leaves_running_sibling_untouched`도 함께 갱신할 것.
- **취소**: blocked/ready/running → failed("사용자가 실행을 중단함"),
  이미 끝난 작업(done)의 출력은 보존.
- **진행률**: 작업 0개면 0.0, done+failed를 끝난 작업으로 집계, 소수 1자리 반올림.

### test_generator.py — 워크플로우 생성 (LLM 경계)

LLM은 신뢰할 수 없는 입력원이므로, 잘못된 출력이 들어와도 시스템이 항상 유효한
그래프를 확보하는지 검증한다. HTTP는 httpx 내장 `MockTransport`로 대체하고
호출 페이로드(모델·온도·메시지)를 기록해 재시도 사다리를 직접 확인한다.

- **순수 파싱 (`_extract_json`)**: 산문에 섞인 JSON, `<think>…</think>` 추론
  블록 제거, 마크다운 코드펜스, JSON 없음/깨진 JSON 시 예외.
- **폴백 (`_fallback_graph`)**: 수집→분석→정리 3단계가 **항상** `validate_graph`를
  통과하는지, 사용자 프롬프트가 role_prompt에 포함되는지, 긴 프롬프트 이름 절단.
- **생성 (`generate_workflow`)**:
  - 정상 응답 → `source="llm"`, id 없는 노드에 `agentN` 부여,
    model/allowed_folders 필드 보강, source/target 빠진 간선 제거.
  - 순환 그래프 응답 → 온도 0.3에서 0.1로 낮춰 1회 재시도 후 폴백.
  - JSON 아닌 잡담 / Ollama 연결 오류(미설치) → 폴백. 어떤 경우에도
    "생성 실패"가 사용자에게 돌아가지 않는다.
  - 1차 실패, 2차 성공 → `source="llm"` 유지.
- **수정 (`revise_workflow`)**: 같은 id로 유지된 노드의 model·allowed_folders·
  position(캔버스 좌표)이 보존되는지, LLM이 비운 name/role_prompt는 이전 값으로
  복원되는지. 생성과 달리 **폴백이 없어서** 실패 시 예외를 던져 사용자의 기존
  그래프를 보존하는지.

### test_security.py — 인증 기반

- **비밀번호**: 해시 왕복, 오답 거부, 같은 입력이라도 해시가 매번 다른지(솔트).
- **토큰 발급**: payload에 사용자 id(`sub`)와 미래의 만료(`exp`)가 담기는지.
- **`get_current_user` 거부 경로** (전부 401): 깨진 토큰, 만료된 토큰,
  **다른 키로 서명된 위조 토큰**, 숫자가 아닌 `sub`, `sub` 누락,
  형식은 유효하지만 삭제된 사용자를 가리키는 토큰.

### test_api.py — API 통합 (E2E)

FastAPI `TestClient`로 라우터·스키마·의존성 주입까지 실제 HTTP 계층을 지나는
테스트. 워크플로우 생성 LLM만 mock으로 대체한다(파싱 검증은 test_generator 담당).

- **해피 패스 1개로 UC-101~206 전 구간**: 회원가입 → 로그인 → 기기 등록(api_key
  발급) → 하트비트(온라인 전환, specs 병합) → 서비스 생성 → 실행 시작 →
  워커가 a→b→c 순서로 `POST /worker/tasks/next` 폴링·결과 제출(b가 받는
  `input_context`에 a의 출력이 들어 있는지 확인) → 작업 소진 시 204 →
  실행 `completed`·진행률 100·말단 결과만 통합 → 이력 목록에 서비스명 표시.
- **인증 가드**: 토큰 없는 `/auth/me` 401, 중복 아이디/이메일 가입 409
  (UC-101 e201/e202), 오답·미존재 계정 로그인 401(e301), 잘못된 기기 키 401.
- **서비스 가드**: 공백 프롬프트 422(UC-201 e201), **순환 그래프 저장 시 422 +
  기존 그래프 보존**(UC-202 e301), `/validate` 응답, 중복 기기명 409(e601).
- **실행 가드**: 온라인 기기가 없으면 409(F2-403), 공백 실행 프롬프트 422
  (UC-203 e301), 취소 후 워커에 작업이 배포되지 않고 재취소는 409.
- **워커 결과 제출 가드**: 작업을 할당받지 않은 기기의 제출 404(가로채기 차단),
  같은 작업 중복 제출 409, `failed` 보고 시 실행이 실패로 전환(UC-204 e202).
- **사용자 격리**: 다른 사용자의 서비스 조회/수정 404, 목록에서 미노출,
  다른 사용자의 기기는 작업을 받지 못하고(204) 실행 이력도 404.

## 워커 테스트 상세

### test_sandbox.py — 폴더 화이트리스트 (요구사항 2.4/2.5)

계약: **서버가 어떤 경로를 지시하더라도** 허용 폴더 밖 접근은 거부된다.
개인정보를 기기 밖으로 내보내지 않는다는 제품 원칙의 마지막 방어선이므로
공격 시나리오 위주로 검증한다.

- **`is_allowed` 차단**: 허용 폴더 밖 경로, `..` 경로 탈출
  (`docs/../secret.txt`), **심볼릭 링크 탈출**(허용 폴더 안의 링크가 밖의 파일
  또는 디렉터리를 가리키는 경우 — `resolve()`가 실제 목적지로 판정),
  접두사 혼동(`/x/docs` 허용이 `/x/docs-evil`을 허용하지 않음), 빈 화이트리스트.
- **`is_allowed` 허용**: 폴더 내부 파일(중첩 포함), 허용 폴더 자체, `~` 확장,
  아직 존재하지 않는 경로(위치 판정만 — 읽기는 read_file에서 실패).
- **`read_file`**: 허용 파일 읽기, 차단 경로/심볼릭 링크/디렉터리/없는 파일 시
  `SandboxError`, 200KB(`MAX_FILE_BYTES`) 절단, **한글 3바이트 문자 중간에서
  잘려도** 예외 없이 대체 문자로 처리.
- **`list_files`**: 텍스트 확장자만(대문자 `.TXT` 포함), 숨김 파일·`.git` 제외,
  `node_modules` 등 `SKIP_DIRS` 미순회, 중첩 디렉터리 순회, `max_entries` 상한,
  존재하지 않는 폴더 건너뛰기, 폴더가 아닌 경로 무시.

### test_executor.py — 작업 실행기

Ollama 호출(`httpx.post`)을 monkeypatch로 대체하고 요청 페이로드를 기록한다.

- **`_chat`**: 응답 공백 제거, qwen3 등 reasoning 모델의 `<think>` 블록 제거,
  요청에 모델·메시지·온도·`stream=False`가 올바르게 실리는지.
- **`run_task`**: 작업에 모델이 지정되면 그 모델, 없으면 기기 기본 모델.
- **`_select_relevant_files`** (컨텍스트 오염 방지): 파일이 1개면 LLM을 호출하지
  않음, LLM이 고른 목록과 실제 파일 목록의 **교집합만** 사용(지어낸 경로 제거,
  원본 순서 유지), LLM 실패·잡담 응답 시 전체 포함으로 폴백.
- **`_build_messages`**: role_prompt가 시스템 메시지로(없으면 기본 프롬프트),
  input_context 포함, 파일 발췌는 `FILE_CONTEXT_LIMIT`(5개)·파일당 4,000자
  상한, 빈 폴더면 파일 섹션 미포함.

## 공용 fixture (`server/tests/conftest.py`)

| fixture / 헬퍼 | 역할 |
|---|---|
| `db` | 인메모리 SQLite(StaticPool) 세션. 테스트마다 새 DB |
| `make_user` / `make_device` / `make_execution` | 모델 팩토리 (서비스+실행 스냅샷 포함) |
| `LINEAR` | a→b→c 그래프 |
| `DIAMOND` | a→b, a→c, b→d, c→d 그래프 (병렬 분기/합류 검증용) |
| `client` (test_api.py) | `get_db` 오버라이드 + 생성 LLM mock을 붙인 TestClient |
| `fake_ollama` (test_generator.py) | `httpx.MockTransport` 기반 Ollama 대체. 호출 페이로드 기록 |

conftest 최상단에서 `MARS_DATABASE_URL=sqlite://`를 설정해 `app.main` 임포트가
테스트 디렉터리에 실제 `mars.db` 파일을 만들지 않게 한다.

## 테스트 작성 시 알아둘 것

- **선점 UPDATE와 세션 캐시**: `claim_next_task`의 원자적 UPDATE는
  `synchronize_session=False`라서, 테스트가 미리 로드해 둔 TaskRecord 인스턴스에는
  결과가 반영되지 않는다. 선점 후 상태를 단언하려면 `db.expire_all()`로 다시
  읽어야 한다 (운영에서는 요청마다 새 세션이라 문제없음).
- **HTTP 헤더는 latin-1만** 허용되므로 헤더 값 테스트에 한글을 넣으면
  UnicodeError가 난다.
- 새 그래프 모양이 필요하면 conftest의 `graph_of(node_ids, edge_pairs)`를 쓰면 된다.

## 아직 다루지 않는 영역

- **웹 프론트엔드** (`web/`) — CI에서 `tsc` 타입체크·빌드만 수행. 컴포넌트
  테스트(vitest)는 미도입.
- **워커 CLI/메인 루프** (`agent/main.py`, `config.py`) — 등록·폴링 루프는
  서버 API 통합 테스트로 프로토콜만 간접 검증.
- **실제 동시성** — 선점은 순차 호출로 검증. 멀티스레드/멀티프로세스 경쟁은
  SQLite WAL 환경에서 별도 부하 테스트가 필요하다.
